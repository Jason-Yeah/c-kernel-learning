# 现有内容

## 基本流程：

主线程`main`在创建`logger`后在后台会立刻启动一个线程`_work_thread`

其中主线程赋值main函数中一条一条语句的执行，工作线程负责执行lambda表达式中逻辑，即`while(_lqueue.pop(msg)) ...`。

## logger构造逻辑

根据logger构造函数：

1. 初始化列表方式初始化文件流`ofstream`打卡文件设置打开方式，初始化退出标志为false
2. 检查文件是否正常打开，不正常抛出异常
3. 启动一个`_work_thread`线程，会一直等待日志队列中的日志信息，没有信息阻塞，有信息就消费并输入到文件流中。

## log逻辑

`log`函数由主线程调用产生日志信息，其逻辑是首先获取当前日志级别通过`enum class log_leve{};`枚举类型获取，在通过`log_queue::push`产生日志消息放入消息队列。主线程只负责产生日志消息到缓冲队列，而logger内部线程负责消费消息，写入到日志文件中

### log_queue::push逻辑

首先为了同步需要加锁（`_queue`可能被多个线程访问，加锁防止数据竞争的问题），通过`std::lock_guard<std::mutex> lock(_mtx);`获取锁同时加锁，此时生产日志信息至缓冲区，如果当前缓冲区信息数为1，唤醒一个消费者进程通过条件变量方式。

### 后台进程挂起阻塞原因

后台进程一直`while(_lqueue.pop(msg)) ...`等待逻辑是pop函数

首先防止数据竞争问题，先获取锁加锁，这里通过`std::unique_lock`方式，接着采用条件变量wait()函数等待排定。

```cpp
_cond_var.wait(lock, [this](){
    return !_queue.empty() || _is_shutdown;
});
```

表示上来先进行lambda表达式判定，只要队列为空且程序不要求关闭，就挂起阻塞，一直卡wait这块，同时释放锁。如果队列不空或程序要关，不再wait走下面消费逻辑，此时被唤醒后还会加锁。

## 析构逻辑

首先把退出标志为true（暂时没用），同时shutdown，然后等后台进程结束，然后关闭文件流。

### shutdown

先加锁，然后`_is_shutdown`标志为true，然后唤醒所有等待进程

## 扩展

## 扩展内容&思路

- 除了文件输出外，支持控制台输出等多种输出目标。
- 使用观察者模式设计灵活的输出组件。
- 日志文件分割：
  - 根据日志文件大小或日期进行日志文件的轮转，防止单个日志文件过大。
  - 在`processQueue`中监控文件大小，达到阈值时切换至新文件。

### 思路

1. 原logger仅持有`std::ofstream _log_file`一个文件流，改为logger持有多个`log_sink`输出目标
2. 由于一条日志可能同时输出到多个地方，不采用策略模式从多个选一个，而是一条日志广播到多个输出目标采用观察者模式，使得多个输出组件能同时接收日志。
3. 文件轮转输出：负责写文件、监控文件大小与当前日期决定是否切换文件
